The extraction schema is a JSON document attached to every scraping target. It tells the agent which fields to extract from the rendered page, how to find them, and how to coerce the raw text into a typed value.

## Top-level structure

```json
{
  "version": "1",
  "js_wait_selector": "[data-loaded]",
  "page_timeout_s": 30,
  "fields": [ ... ]
}
```

| Property | Type | Required | Description |
|---|---|---|---|
| `version` | `"1"` | ✓ | Schema version. Must be the string `"1"`. |
| `fields` | array | ✓ | 1–100 field definitions. |
| `js_wait_selector` | string (max 500 chars) | — | CSS selector the agent waits for before starting extraction. Useful for pages that lazy-load content after the initial render. |
| `page_timeout_s` | integer (5–120) | — | Hard timeout in seconds for the full page load, including the `js_wait_selector` wait. Defaults to 30 seconds if omitted. |

## Fields

Each item in `fields` defines one key in the result JSON.

```json
{
  "name": "price",
  "type": "number",
  "required": true,
  "extractors": [ ... ],
  "transform": { "regex": "([\\d.,]+)" }
}
```

| Property | Type | Required | Description |
|---|---|---|---|
| `name` | string | ✓ | Result key. Lowercase letters, digits, underscores. Must start with a letter or underscore. Max 64 chars. |
| `type` | string | ✓ | Target type: `"string"`, `"number"`, `"boolean"`, or `"array"`. |
| `item_type` | string | if `type` is `"array"` | Element type for array fields: `"string"`, `"number"`, or `"boolean"`. |
| `required` | boolean | — | If `true`, the job fails when no extractor yields a value. Default: `false`. |
| `extractors` | array | ✓ | 1–10 extractor strategies tried in order. First non-empty value wins. |
| `transform` | object | — | Post-processing applied after extraction. |

## Extractor strategies

The agent tries each extractor in the `extractors` array in order and uses the first one that returns a non-empty value. Each extractor must have a `strategy` field that acts as its discriminator.

### `css`

Runs a CSS3 selector against the fully rendered DOM.

```json
{
  "strategy": "css",
  "selector": "h1.product-title",
  "attribute": "innerText",
  "multiple": false
}
```

| Property | Required | Description |
|---|---|---|
| `selector` | ✓ | CSS selector expression (max 1000 chars). |
| `attribute` | — | What to extract from the matched element. `"innerText"` (default) returns the visible text content. `"html"` returns the inner HTML. Any other value is treated as an HTML attribute name, e.g. `"href"`, `"src"`, `"content"`, `"data-price"`. |
| `multiple` | — | Collect all matching elements into a list. Pair with `type: "array"` fields. Default: `false`. |

### `xpath`

Evaluates an XPath 1.0 expression against the DOM.

```json
{
  "strategy": "xpath",
  "selector": "//span[@itemprop='price']/@content",
  "attribute": "innerText",
  "multiple": false
}
```

| Property | Required | Description |
|---|---|---|
| `selector` | ✓ | XPath 1.0 expression (max 1000 chars). |
| `attribute` | — | Same semantics as the CSS `attribute` property. Ignored when the XPath expression itself selects an attribute node (e.g. `/@content`) or text node (e.g. `/text()`), since the node value is used directly. |
| `multiple` | — | Collect all matched nodes. Default: `false`. |

### `schema_org`

Finds a JSON-LD block whose `@type` matches, then traverses a dot-notation path into it.

```json
{
  "strategy": "schema_org",
  "schema_type": "Product",
  "path": "aggregateRating.ratingValue"
}
```

Matching is case-insensitive and handles both scalar (`"@type": "Product"`) and array (`"@type": ["Product", "Thing"]`) type values. Pages that wrap their JSON-LD in a `@graph` container are also supported: the agent searches each item in the graph.

When you don't know the `@type` in advance, use `"*"` to search all JSON-LD blocks and return the first match that contains the given path:

```json
{
  "strategy": "schema_org",
  "schema_type": "*",
  "path": "aggregateRating.ratingValue"
}
```

| Property | Required | Description |
|---|---|---|
| `schema_type` | ✓ | Schema.org type to match. Case-insensitive. Examples: `"Product"`, `"Offer"`, `"AggregateRating"`, `"Article"`. Use `"*"` to match any type. |
| `path` | ✓ | Dot-notation path into the matched object. Array indexing via numeric keys: `"images.0"`. |

### `microdata`

Queries elements with a matching `itemprop=` attribute per the HTML Microdata specification.

```json
{
  "strategy": "microdata",
  "itemprop": "ratingValue",
  "attribute": "content",
  "multiple": false
}
```

The default extraction behavior depends on the element type:
- `<meta itemprop="...">`: returns the `content=` attribute
- `<link itemprop="...">`: returns the `href=` attribute
- All other elements: returns `innerText`

Set `attribute` explicitly to override this default for any element type.

| Property | Required | Description |
|---|---|---|
| `itemprop` | ✓ | Value of the `itemprop=` attribute to match. |
| `attribute` | — | HTML attribute to extract. Overrides the default element-based behavior described above. |
| `multiple` | — | Collect all matching elements. Default: `false`. |

### `meta`

Extracts the `content=` value from a `<meta>` tag by `property=` or `name=` attribute.

```json
{
  "strategy": "meta",
  "property": "og:title"
}
```

| Property | Required | Description |
|---|---|---|
| `property` | ✓ | Matches `property=` first, then `name=` (case-insensitive). Examples: `"og:title"`, `"og:image"`, `"twitter:card"`, `"description"`. |

### `json_ld`

Raw access to a JSON-LD block without `@type` matching. Use `schema_org` when the type is a known Schema.org type.

```json
{
  "strategy": "json_ld",
  "path": "mainEntity.offers.0.price",
  "block_index": 0
}
```

| Property | Required | Description |
|---|---|---|
| `path` | ✓ | Dot-notation path from the root of the JSON-LD block. Examples: `"@graph.0.name"`, `"mainEntity.offers.0.price"`. |
| `block_index` | — | Zero-based index selecting which `<script type="application/ld+json">` block to read when a page has multiple. Default: `0`. |

## Transform

Optional post-processing applied to the extracted string before type coercion. Steps are applied in this order: **trim → regex → prepend → append**.

```json
{
  "trim": true,
  "regex": "([\\d.,]+)",
  "prepend": "https://example.com",
  "append": " USD"
}
```

| Property | Description |
|---|---|
| `trim` | Strip leading/trailing whitespace. Default: `true`. Applied first. |
| `regex` | Regular expression. If the pattern contains a capture group, the content of the **first** capture group is used. If there is no capture group, the full match is used. Extraction fails if the pattern does not match. |
| `prepend` | String prepended to the value after trim and regex. Useful for constructing absolute URLs from relative paths. |
| `append` | String appended to the value after trim and regex. |

## Complete example

```json
{
  "version": "1",
  "js_wait_selector": "[data-product-loaded]",
  "fields": [
    {
      "name": "title",
      "type": "string",
      "required": true,
      "extractors": [
        { "strategy": "schema_org", "schema_type": "Product", "path": "name" },
        { "strategy": "css", "selector": "h1.product-title" },
        { "strategy": "meta", "property": "og:title" }
      ],
      "transform": { "trim": true }
    },
    {
      "name": "price",
      "type": "number",
      "required": false,
      "extractors": [
        { "strategy": "schema_org", "schema_type": "Offer", "path": "price" },
        { "strategy": "microdata", "itemprop": "price", "attribute": "content" },
        { "strategy": "css", "selector": ".price-now" }
      ],
      "transform": { "regex": "([\\d.,]+)" }
    },
    {
      "name": "images",
      "type": "array",
      "item_type": "string",
      "extractors": [
        { "strategy": "css", "selector": ".gallery img", "attribute": "src", "multiple": true }
      ]
    }
  ]
}
```

<Callout type="info">
  The extraction schema is validated against a JSON Schema (draft 2020-12) when a template is created or updated. The Monaco editor in the orchestrator UI provides autocompletion and inline error highlighting based on the same schema.
</Callout>
